# Contributing to OntoVerse

Thank you for your interest in contributing to Ontoverse! We welcome contributions from the community to help improve and expand the project. Please follow the guidelines below to ensure a smooth contribution process.

## How to Contribute

### 1. Reporting Issues
If you encounter a bug, have a feature request, or want to suggest improvements, please open an issue:
- Navigate to the [Issues](https://github.com/AstraZeneca/ontoverse-kg-choreographer/issues) tab in the repository.
- Click on **New Issue**.
- Provide a clear and descriptive title.
- Describe the problem, expected behavior, and any relevant steps to reproduce it.
- Attach screenshots or logs if applicable.

### 2. Submitting Pull Requests (PRs)
We welcome pull requests for bug fixes, new features, and documentation improvements. Please follow these steps:

1. **Fork the Repository**: Click the “Fork” button at the top right of the repo.
2. **Clone Your Fork**:
   ```sh
   git clone https://github.com/AstraZeneca/ontoverse-kg-choreographer.git
   cd ontoverse
   ```
3. **Create a Branch**: Use a meaningful branch name related to your change.
   ```sh
   git checkout -b feature-or-fix-name
   ```
4. **Make Your Changes**: Ensure that your code follows the project's guidelines and is well-documented.
5. **Commit Your Changes**: Use descriptive commit messages.
   ```sh
   git commit -m "Add feature/fix: Brief description"
   ```
6. **Push to Your Fork**:
   ```sh
   git push origin feature-or-fix-name
   ```
7. **Open a Pull Request**:
   - Go to the [Pull Requests](https://github.com/AstraZeneca/ontoverse-kg-choreographer/pulls) section of the main repo.
   - Click **New Pull Request** and select your fork and branch.
   - Provide a clear description of the changes.
   - Mention related issues (e.g., “Fixes #42”).
   - Submit the PR for review.

## Code Style and Best Practices
- Follow existing code conventions and formatting.
- Include tests for new features or fixes whenever possible.
- Ensure that the code is properly documented.
- Keep PRs focused and avoid bundling unrelated changes.

## Review Process
- The project maintainers will review PRs as soon as possible.
- Maintain an open dialogue with reviewers and be open to feedback.
- PRs that meet the quality standards will be merged into the main branch.
